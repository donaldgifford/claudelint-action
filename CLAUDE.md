# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A **composite GitHub Action** that wraps the [`claudelint`](https://github.com/donaldgifford/claudelint) CLI. The Action itself contains no application source — the entire implementation is `action.yml`. The actual linter lives in a separate repo; this Action downloads a released binary from there, verifies its SHA-256, and runs it.

## Architecture

`action.yml` is the entire product. It defines three composite steps that run in order:

1. **Resolve version** (`steps.resolve`) — if `version: latest`, calls the GitHub API for `donaldgifford/claudelint`'s latest release tag; otherwise uses the literal input.
2. **Download claudelint binary** (`steps.download`) — maps `RUNNER_OS`/`RUNNER_ARCH` to the matching release archive (`claudelint_${os}_${arch}.tar.gz`, or `.zip` on Windows), downloads `checksums.txt` from the same release, verifies the archive with `sha256sum --check`, extracts, and installs to `${HOME}/.claudelint/bin` (added to `$GITHUB_PATH`).
3. **Run claudelint** (`steps.run`) — invokes `claudelint run` with the input flags. Always re-runs once with `--format=json --max-warnings=-1` to a temp file purely to extract `diagnostic_count` and `severity_count.error` via `jq` for the action outputs, regardless of the user-selected format.
4. A conditional fourth step uploads SARIF to Code Scanning via `github/codeql-action/upload-sarif@v4` when `upload-sarif=true` and `format=sarif`.

When changing input/output names, update both the `inputs:`/`outputs:` schema **and** the `README.md` tables (they are hand-maintained, not generated).

## Testing

The Action self-tests by using itself against `fixtures/ok/` in `.github/workflows/test.yml`. There is no local test harness — to validate changes you push and let the matrix (`ubuntu-latest`, `macos-latest`) run. The clean fixture must always produce zero errors; the assertion step fails the workflow otherwise. A separate `test-sarif-upload` job exercises the SARIF upload path.

When adding fixtures, mirror the `fixtures/ok` pattern. Anything you add becomes part of the Action's test signal.

## Tooling (via mise)

Tool versions are pinned in `mise.toml`. After `mise install`, available commands:

- `actionlint` — lint `action.yml` and `.github/workflows/*.yml` (config in `.github/actionlint.yml`).
- `yamlfmt` / `yamllint` — YAML formatting and lint (configs in `.yamlfmt.yml`, `.yamllint.yml`).
- `markdownlint-cli2` / `prettier` — Markdown lint and formatting (`.markdownlint.yaml`, `.prettierrc.yaml`).
- `git-cliff` — regenerate `CHANGELOG.md` from conventional commits (see below).

There is no `justfile`/`Makefile` despite `labeler.yml` referencing one — that file is template boilerplate from the scaffolder.

## Changelog workflow

`CHANGELOG.md` is **generated**, not hand-edited. Two workflows enforce this:

- `.github/workflows/changelog.yml` — on PR, regenerates and diffs against committed `CHANGELOG.md`; PR fails if stale.
- `.github/workflows/changelog-regen.yml` — on push to `main` (skipped for `chore` commits), regenerates and auto-commits as `chore(changelog): Auto-sync` using the default `GITHUB_TOKEN`.

To update locally: `git-cliff -o CHANGELOG.md`. Commit-message style is conventional commits, parsed by `cliff.toml` (groups: feat, fix, refactor, docs, perf, style, test, chore/ci/build, security, revert).

## Scaffold lock

`.forge-lock.hcl` tracks files generated by the `forge` scaffolder (`std/new` blueprint). Files listed with `strategy = "overwrite"` (`.markdownlint.yaml`, `.prettierrc.yaml`, `.yamlfmt.yml`, `.yamllint.yml`, `.docz.yaml`, etc.) will be regenerated on the next forge sync — edit upstream in the registry if you need permanent changes, otherwise expect overwrites.
